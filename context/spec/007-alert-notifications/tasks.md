# Tasks: Alert Notifications

- **Functional Spec:** [functional-spec.md](./functional-spec.md)
- **Technical Spec:** [technical-considerations.md](./technical-considerations.md)
- **Stack:** Go std-lib dashboard (`dashboard/`) — new `internal/notify` + `internal/alerts`, poller extension, `main.go` config, Overview/Events templates + `/api/alerts`. **Pure-app change — no Terraform/IAM/network** (see decision below). All sub-tasks → `go-fullstack`.

**Confirmed decisions** (resolved spec `[NEEDS CLARIFICATION]` + owner choices this session):

- **Config = environment variables only.** The webhook URL (`DASHBOARD_WEBHOOK_URL`) and all thresholds are read from the environment at startup. **No SSM, no Terraform/IAM** — overrides the technical spec's Option-A (SSM SecureString) recommendation so the dashboard stays portable across clouds/VPS. How the env var is set on the host (systemd `EnvironmentFile`, container env, VPS secret store) is the operator's concern, out-of-band. **Empty URL → alerting silently disabled** (evaluator still runs for the in-UI view; notifier is a no-op) — opt-in per functional spec §2.2.
- **Delivery = Slack-compatible incoming webhook, interface-first.** v1 POSTs a `{"text":…}` body to the configured URL (works on Slack-App webhooks, Mattermost, Discord `…/slack`, Google Chat). Delivery is isolated behind a `Notifier` interface so a Slack-bot / Telegram / ntfy / PagerDuty transport is a later drop-in with **zero change to the evaluator**. No bot/OAuth in v1.
- **Five watched conditions:** service-down, high-disk (90%), sustained-CPU (90% / 5min), **peer-down** (kept) and **per-peer cumulative transfer** (NEW). Peer-down uses the **seen-online-once gate** (the spec's proposed default — a client must have been online at least once this run before it can fire). Per-peer cumulative-transfer is **added alongside** peer-down (not a replacement).
- **Thresholds/windows** (all env-configurable, documented defaults): disk 90%, CPU 90% over a 5-min sustain window, peer-stale 10min, **cooldown 30min**. Transfer cap default to be set in Slice 4 (e.g. 50 GB Rx-or-Tx since dashboard start).
- **UI placement:** active-alerts strip/card atop the **Overview** tab + alert entries in the **Events** tab + an "alerting enabled/disabled" status. No new tab.

**Grounding (existing seams):** the poller already gathers everything an alert needs each tick — `wg-quick@wg0` status (systemd reader), CPU & disk (`system_metrics`), and per-peer cumulative counters (`client_traffic.rx/tx_bytes_cum`) + handshake recency. The evaluator is **pure given inputs + an injected clock** (fully unit-testable); state is **in-memory** (no alert DB) and re-arms from current state on restart (a still-bad condition may notify once post-restart — accepted, §2.3). Delivery runs **off the poll critical path** so a slow webhook never blocks the tick. Hard constraint preserved: **no new inbound surface, no auth change** — alerting adds only outbound HTTPS.

Each slice leaves the dashboard runnable; in-session proof is `go test` + httptest + a fake HTTP transport. Real webhook receipt (a green build does **not** prove delivery) is the manual Slice 6.

  > **Implementation notes (Slice 2):** New pure pkg `internal/alerts` — `Evaluator` (`New(Config{Cooldown,Host})`, `Evaluate(now, Input) []Event`) over an unexported condition-agnostic `stateMachine.observe(firingNow, now, cooldown) (Kind, bool)` funnel that owns ALL edge-trigger/cooldown/recovery/re-arm logic in one place. `OK→FIRING` emits one `Fire`; `FIRING→OK` emits one `Recovery` then re-arms; still-firing suppresses until `now.Sub(lastFire) >= cooldown` (at-boundary re-notifies = a reminder), `DefaultCooldown=30m` (Config field, env knob deferred).
