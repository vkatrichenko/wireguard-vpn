# Tasks: Dashboard Design System & Responsive Refresh

- **Functional Spec:** [functional-spec.md](./functional-spec.md)
- **Technical Spec:** [technical-considerations.md](./technical-considerations.md)
- **Stack:** Go std-lib dashboard (`dashboard/`) — CSS-led (`web/static/app.css`), embedded fonts (`web/static/fonts/`), minimal template/class edits across the 6 tabs + ~25 card fragments, `charts.js` theme bridge. **No rendering-model change** (server-rendered `html/template` + htmx + Chart.js). All sub-tasks → `go-fullstack`, **applying the `frontend-design` skill** for the aesthetic work.

**Confirmed decisions** (from the 009 design):

- **Refined & distinctive** — a cohesive, characterful-but-professional design system; legibility/density paramount (ops tool). Recommended direction: "precision instrument" (confident monospace for technical data, precise grid, calm depth).
- **Embed a distinctive font** — self-hosted, subset `woff2` (display + body + mono), vendored via the existing `go:embed all:web`, OFL/permissive, recorded in `VENDORED.txt`. No external font request.
- **Fluid across all sizes** — `clamp()` type/space + `@container` queries (with `@media` fallback), ~360px phone → ultrawide; no clipping, no page h-scroll ≥360px, no ultrawide dead space.
- **Whole dashboard** — one design-token system applied across all 6 tabs + every card.

**Grounding (current state):** `app.css` (~1,168 lines) is hand-written with numbered sections, a `system-ui` font stack, a `:root` + `[data-theme="dark"]` token block, and **only one `@media (max-width:600px)`** + one `prefers-reduced-motion` block — no `@layer`/`@font-face`/`clamp()`/`@container`. Templates already use semantic classes (`.card`, `.status-pill`, `.client-table`, `.overview-grid`, …), so most of the refresh is CSS-only. `charts.js` already reads CSS vars (006). `body { max-width:1400px }`. **Hard constraints:** fully offline, no SPA/build step, dark mode, functional parity (design-only), preserve the 006 geo-map 2:1 aspect/projection + timeline + charts.

**Verification reality:** there is **no browser MCP**, so visual + responsive correctness is an **operator (manual) gate** on each slice. Automated proof per slice = existing render/partial tests stay green (structure unchanged), offline-URL grep, `gofmt`/`go vet`, `make build` (note binary-size delta). Each slice leaves the dashboard building and rendering.
