# Tasks: Dashboard Binary via GitHub Releases

- **Functional Spec:** [functional-spec.md](./functional-spec.md)
- **Technical Spec:** [technical-considerations.md](./technical-considerations.md)
- **Stack:** Go dashboard (`dashboard/`, ldflags), GitHub Actions (`.github/workflows/`), Terraform (`terraform/dev/` + `terraform/modules/{wireguard,dashboard,github-oidc}`), EC2 user-data (`terraform/modules/wireguard/templates/user-data.txt`).

Each slice leaves the tree in a **validate-able, runnable** state. In-session proof is `go test` / `go build` + run (Slice 1), workflow static-lint + reasoning (Slice 2), `terraform validate` + `plan` + a shell test of the checksum logic (Slices 3‚Äì4). The real boot/fetch and the tag-triggered release require an operator **apply** + a **public repo** + a **tag push** ‚Äî those are the manual Slice 5, honestly out of in-session reach (no `terraform apply`, pushes are gated, repo-public is a separate precondition).

> **Sequencing & precondition.** Slices 3‚Äì5 only work once the repository is **public** (anonymous asset download) and a release **actually exists** for the pinned tag (Slice 2 must have produced one). Apply Slices 3 + 4 in a **single** `terraform apply` (the curl-from-release path replaces the `aws s3 cp` path in the same change that removes the S3/IAM it depended on). Slices 1‚Äì2 are pure app/CI and land independently of any apply.
